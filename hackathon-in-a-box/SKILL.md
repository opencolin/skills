# Hackathon in a Box

Turn a theme, sponsor, or rough concept into a complete hackathon operating plan that an organizer could run from top to bottom. This is operational, not inspirational. Assume the user has to actually execute it.

## Operating principles

A hackathon succeeds or fails on logistics and clarity, not on the prize size. Every output should reduce the organizer's cognitive load on the day. Be specific about timing, ownership, and decision points. Default to a one-day format unless the user specifies a weekend or multi-day event, and scale the plan accordingly.

Write in a clear operator voice. No emojis. No em dashes. Mark unknowns as `[assumption: ...]` and keep moving.

## Required output structure

Produce all sections in this order.

### 1. Event summary
Three lines: the theme, the target participant, and the single outcome the event is optimizing for (recruiting, ecosystem adoption, content, community growth). The outcome shapes every other decision, so state it explicitly.

### 2. Run of show
A minute-by-minute or block-by-block schedule covering: check-in, kickoff and rules, team formation, build time, checkpoints, submission deadline, demos, judging, and awards. For each block include the time, what happens, and who owns it. Build in buffer. Call out the two moments that most often go wrong (submission cutoff and demo logistics) and how to de-risk them.

### 3. Judging rubric
A scoring rubric with 4 to 5 weighted criteria (e.g., technical execution, use of the sponsor tech, originality, demo quality, real-world potential). Give each a weight and a one-line description of what a high score looks like. Include the scoring scale and how ties break.

### 4. Prize and sponsor tier structure
Prize breakdown by placement, plus a sponsor tier menu (e.g., Title, Track, Supporting) with what each tier gets: branding, judging seat, prize naming, demo access, attendee data rights. Keep sponsor benefits concrete so they map to real money.

### 5. Judge brief
A short, copy-paste brief to send judges before the event: what they are judging, the rubric, the time commitment, how demos will run, and what is expected of them. Judges who arrive unprepared slow everything down, so this brief earns its place.

### 6. Participant comms timeline
Messages to send participants: registration confirmation, T-minus 1 week (what to prep, what to bring, API keys to set up in advance), day-of arrival info, and a post-event recap and thank-you with next steps. Pre-event technical setup is the highest-leverage message because it saves hours on the day.

### 7. Logistics checklist
A scannable checklist grouped by: venue and access, food and power, wifi and dev environment, hardware and credentials, signage, and AV for demos. This is the thing the organizer prints and walks the floor with.

## Format note

Make the run of show and the logistics checklist genuinely usable as standalone documents, since organizers often hand these to volunteers.

## Quality bar

Before returning, check: could a first-time organizer run this event without having to invent anything themselves? Every decision point should already be made or clearly flagged as a choice.
